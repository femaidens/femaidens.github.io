Configuring GitHub username and email (not required)

You can set the GitHub username and email so that when you push to GitHub, your account is displayed as the account making changes to the repository.

FOR WINDOWS USERS

  • Setting and resetting the GitHub username and email is done in the exact same way.
    1. Open Git Bash.
    2. Type:
      1. git config --global user.name “[enter your name here]”
      2. git config --global user.email “[enter your email here]”
  • For example:
    1. git config --global user.name “Jillian Chong”
    2. git config --global user.email “chongj@bxscience.edu”

FOR MACOS USERS

  1. Open a new bash terminal window in VSCode
    • If the name of the window doesn’t immediately read “bash”, select the drop down menu next to the plus sign and select bash
  2. Follow Step 2 from above.

Pushing Changes to Github

git status
git add -A // -A flag adds ALL modified files to staging area*
git commit -m "enter message here"
git push origin <branch>

Staging area: the area before the code is pushed to GitHub; you can add all modified files or just specific files.

  • git add Drivetrain.java → adds just the Drivetrain file.

Pulling Changes from Github

Pulling changes from GitHub updates your local repository with the changes pushed in the remote repository. To pull changes from GitHub, type:
git pull origin <branch>

Github Branching: Make a New Branch

When working on the robotics repository, we make a branch for each mechanism (including drivetrain). The main branch (formerly called the master branch) is the default branch on GitHub. It should just contain Main.java, Robot.java, Ports.java, Constants.java, and RobotContainer.java. Files in the main branch should be empty (except for code that all other branches share) until other branches are merged into main.

To make a new branch, type the following:
git branch <branch-name>
To switch between branches, type the following (quotation marks are not necessary around the branch name):
git checkout <branch-name>
Keep in mind that git branch does NOT check you out to your new branch. To switch into your new branch, you must follow with git checkout <branch-name> . However, there is a shortcut that combines the two commands:
git checkout -b <branch-name>

Set Up Tracking Branches

Read the following section for what tracking means!
Once you have made a new branch, push it to the remote GitHub repository. It’s the same process as pushing changes to GitHub (described above). HOWEVER, simply pushing the new branch to the remote repository does NOT automatically set up a tracking branch between your local and remote branches. To set up your local branch to track the remote branch, run the following command with the remote as origin
git branch -u <remote/branch-name>


Or, you can set up the tracking branch right after creating it by pushing it with the following command:
git push -u origin <branch-name>


To check which local branches are tracking remote branches (if any at all), run the following command. It will also provide information about how many commits ahead or behind your local branch are compared to the remote branch.
git branch -vv

The branches that appear on GitHub (remote repository) may not all be on your local version of the code (on your computer). To check all existing branches in the remote repository (on GitHub), use the following command:
git branch -a

To edit (checkout) to a branch that exists in the remote repository but not your local repository, run:
git checkout <branch-name>
This will automatically track your local branch to your remote branch.

Concepts in Context

To edit an existing remote branch on your own local repository, first checkout to your desired branch then run the command git pull. For example, if the intake subsystem exists in the remote repository but not your local repository, you would run the following commands:
git pull
git branch -a // see all existing branches in the remote repository
git checkout intake // checks out to the intake branch

Why do we make a branch for each subsystem?

  • When we test mechanisms, we usually test each mechanism individually. By having only one subsystem on a branch, you can just test the subsystem on the branch and eliminate possible areas of error (ex. code from other subsystems that may be interfering with testing).
  • When we run the robot during competition, we combine all the branches into one final branch or project that is run during comps. This final branch should have all the subsystems, one Main.java, Robot.java, RobotContainer.java, Ports.java, and Constants.java.

Github Branching: Merging Branches

Merging branches is an essential part of code collaboration.

Managing Merge Conflicts

In VS Code:

  • Resolve in Merge Editor will show up in bottom right corner
  • Choose which code you want to keep (either Incoming or Current). A new tab on the bottom will show the version you want to keep.

In Github

  • After approval for the Pull Request, select Resolve Conflicts
  • Will bring you to the GitHub code editor. Sections that require attention will have their line numbers highlighted
  • After changes are made, Mark as Resolved in the top right corner will light up
  • After all files are marked as resolved, merging can happen
  • Make sure that file has no errors by visiting other files (via VS Code)

Resolving Github Errors

Given the annoyingness of GitHub, you’re likely to encounter errors. The best way to resolve an error you’re unsure of is to Google it!
If you cloned your repository wrong, delete the local version on your computer and reclone it from the remote repository online.
